# Engineering Decisions

This document exists for one reason: so that "why did you build it that way?" has a real answer instead of an improvised one. Every decision below was made deliberately, is defended in `DESIGN.md` and the code itself, and is still true as of the current codebase — not aspirational.

## Why deterministic analysis comes first

The core design bet of this project is that the engineering-check engine has to be substantial and useful **on its own**, with the AI layer switched off entirely. `app/analysis/` implements 28 checks across 9 categories (routing, power, ground, differential pairs, decoupling, placement, manufacturability, thermal, signal integrity) as independent Python modules with named, documented thresholds — no machine learning, no LLM call, no network dependency. Run `pcbinsight review <path>` with no `ANTHROPIC_API_KEY` set and it still produces a full scored report.

This ordering wasn't incidental — it was Phase 0 through Phase 2 of the build, completed and tested before a single line of AI-integration code existed (see `DESIGN.md`'s phase plan). The reason is structural, not sequencing preference: if the deterministic engine were weak, the AI layer would inevitably become the thing doing the real work, and the project would quietly become "an LLM wrapper with better branding" — precisely the outcome `DESIGN.md`'s Constraints section rules out by name. Building the check engine first, proving it against real boards, and only then adding narration on top means the AI can never become load-bearing, because by the time it exists there's nothing left for it to be load-bearing *for*.

## Why the AI cannot invent findings

Claude never runs analysis. It receives a structured digest (`app/ai/summarizer.py`) of findings the deterministic engine already produced and is instructed, in `REVIEW_SYSTEM_PROMPT` (`app/ai/review.py`), to synthesize and prioritize — not to discover new problems:

> "You are acting as a senior PCB design reviewer. You are NOT performing design analysis — that has already been completed deterministically... Do not invent problems or recommendations that are not supported by the supplied data."

The digest itself enforces the same boundary structurally, not just by instruction: it never contains raw geometry (trace coordinates, pad positions, copper polygons) — only aggregate statistics, the evidence block, and the issue list the engine already computed. There is no raw-board information for Claude to analyze even if a prompt injection or a model quirk pushed it to try. This split — deterministic engine finds problems, AI narrates and prioritizes them — is the single decision the whole project's credibility rests on, and it's why the operating name for this feature in `DESIGN.md` is the **AI Engineering Review Layer**, not "AI integration": the name itself states the boundary.

## Why citations are required, and checked in code

Every `Issue` gets a stable, category-prefixed ID (`PWR-004`, `SIG-003`, ...), assigned once per report by the orchestrator (`app/analysis/__init__.py`'s `run_all_checks`) so numbering stays consistent regardless of which checks fired. The system prompt requires Claude to cite these IDs when discussing specific findings and to hedge anything below 0.5 confidence rather than state it as fact.

A prompt instruction alone is not a guarantee — models can still cite a plausible-looking ID that doesn't actually exist in the digest, especially one it invented by analogy to an example shown earlier in its own instructions. `find_unsupported_citations()` closes that gap in code: it regex-extracts every issue-ID-shaped token from Claude's output and diffs it against the digest's real issue IDs, logging a warning on any mismatch. This isn't hypothetical — it caught a real bug during live validation (see the section below). The pattern here — a prompt-level instruction backed by a code-level check that doesn't trust the model to have followed it — is applied twice in this codebase (citations, and the raw-geometry boundary above) and is the general shape of how AI is integrated everywhere in this project: defense in depth, not prompt-trust.

## Why KiCad is parsed directly

`backend/app/parser/sexpr.py` is a hand-rolled S-expression tokenizer/parser, not a wrapper around KiCad's own `pcbnew` Python bindings. Two reasons converged on this, one practical and one deliberate: `pcbnew` bindings weren't even installable in the dev environment at the time (see `DESIGN.md`'s Open Questions), which settled the question in practice — but independent of that, a hand-rolled parser means the tool has zero KiCad/CAD installation dependency at all. Anyone can run `pcbinsight review` against a `.kicad_pcb` file with nothing but Python installed. It also means the parsing logic is original engineering work worth discussing in its own right, not "I imported a library and called a function" — the tokenizer, the S-expression tree walker, and the KiCad-format-specific extraction logic (`kicad_project.py`) are all things this project actually built, verified now against real files including deliberately hostile ones (see `parser_edge_cases` in the validation corpus below).

## Why the Engineering Validation Corpus exists

A tool whose entire pitch is "trust the deterministic engine's output" needs more evidence than a single "does it basically work" fixture. `examples/` is ten purpose-built KiCad boards (`docs/VALIDATION.md` has the full breakdown), each authored to prove one specific thing: `flawed_reference` proves every check category fires when it should; `high_quality_reference` proves the engine doesn't invent problems when there aren't any (100/100, zero findings — the harder property to demonstrate, since a tool that only ever finds things looks broken in a different way); `rf_clock_board`, `power_supply_board`, `thermal_reference`, and `manufacturing_reference` each isolate one category so a change to that category's thresholds can be verified in isolation; `parser_edge_cases` stresses the tokenizer itself (unicode, escaped quotes, hostile characters) rather than the analysis engine; `mixed_realistic_board` exists because "everything is either perfectly clean or a disaster" is not what real boards look like, and a review tool that's only ever tested on extremes doesn't tell you much about ordinary use.

Every board in the corpus is checked by a parametrized regression test (`test_corpus_board_parses_and_scores_without_error`), so the corpus isn't just documentation — it's a live guard against the parser or analysis engine silently regressing.

## Why AI validation is separate from CI

`backend/scripts/validate_ai.py` runs the real review and chat prompts against the live Anthropic API and is explicitly **not** part of the GitHub Actions test suite. Two reasons: it costs real money on every run (it makes ~10 billed API calls), and CI needs to be something that runs on every push without a human deciding whether to spend anything. The unit test suite (`backend/tests/test_review.py`, `test_api.py`) proves the AI layer's *plumbing* — prompt construction, the citation-checking function, the graceful-degradation-without-a-key path — using a dependency-injected fake Anthropic client, which costs nothing and runs on every commit.

But plumbing tests can't tell you whether the *actual model* respects the boundaries it's instructed to respect — whether it really does hedge low-confidence findings, whether it really doesn't invent problems on a clean board, whether adversarial net names really don't break the prompt. That requires a live model. Keeping this as a separate, manually-run script with committed evidence (`reports/ai_validation.{md,json}`) gets both: CI stays free and fast, and there's still a real, reproducible, re-runnable record that the AI layer was validated against actual model output — including the one real bug it caught (the model reused a hardcoded example issue ID from its own system prompt; fixed by removing concrete example IDs from the prompt text, then re-validated with zero hallucinated citations).

## Why transparency is emphasized over benchmark scores

The Engineering Score is a severity × confidence weighted deduction from 100 (`app/analysis/scoring.py`) — named constants, no learned weights, no black-box model. "How was this score computed" has a one-line answer, and that answer is the same for every board, every time. This was a deliberate trade against a more sophisticated scoring approach (a trained model calibrated against real fab yield data, say) that would likely correlate better with real-world outcomes but couldn't be explained in a sentence.

The same instinct shows up in what the project chose *not* to claim: no coverage badge (no live coverage tooling behind it to justify one), no "time saved" metric (no deterministic basis — estimating it would mean fabricating a number to support a story), no manufacturability checks for silkscreen-over-pad or copper-sliver detection (the board model doesn't capture the geometry needed to check them correctly, so the check is left out entirely rather than faked with something that can never actually fire — see `manufacturability.py`'s own docstring). A project whose whole pitch is transparency loses that pitch the moment one of its own numbers turns out to be unverifiable. Every metric in the README and every score this tool produces is something you could ask "how was this computed" about and get a real answer, not a hand-wave.
