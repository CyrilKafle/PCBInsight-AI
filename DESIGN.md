## Power Budget

A single hobby servo can draw 500mA-1A+ under stall/load; a small FPGA dev board's onboard USB/regulator (typically good for well under 500mA total) cannot power even one loaded servo, let alone multiple. **The servo rail must be powered by a separate external 5-6V supply, not the FPGA board's own power.** PCB rev 1 must include a dedicated power input header/connector for this rail, appropriately sized bulk decoupling capacitors near the servo connector, and a common ground between the FPGA board and the servo supply. This is a Phase 1 hardware requirement, not an afterthought — add it to the PCB schematic from the start.

**Phase 1 parts inventory (2026-07-10):** the user already owns an ELEGOO UNO Super Starter Kit, which covers most of the breadboard-stage bill of materials directly: breadboard(s), jumper wires, an SG90 hobby servo, a potentiometer (used for position feedback), and — critically — a breadboard power supply module plus a 9V battery+DC clip, which together *are* the external 5-6V servo supply called for above. Only two new parts need to be purchased for Phase 1: an FPGA dev board (Tang Nano 9K class) and an SPI ADC (MCP3208 or MCP3008 — Arduino kits never include one since Arduino has built-in analog inputs). Estimated new spend: ~$20-45, comfortably under the original ~$100 budget. Wiring note: the ADC must be powered from the FPGA board's 3.3V rail, not the kit's 5V breadboard rail, so its SPI logic levels match the FPGA's 3.3V I/O — the servo itself doesn't need level-matching since it only receives a PWM signal, not bidirectional logic.

## Recommended Approach

**Approach B (full multi-joint arm), executed in three gated phases, with Approach A as the literal first phase and Approach C as an optional stretch phase if time remains:**

**Phase 0 — Simulate first (no purchases).** Design the single-joint closed-loop control datapath in SystemVerilog: PWM generator module, SPI ADC / position-feedback read module, and a P or PID controller closing position error to a target. Validate entirely in simulation (Icarus Verilog or Verilator + a testbench, viewed in GTKWave) — prove correct PWM timing and correct closed-loop behavior against a simulated feedback signal before any hardware exists. This is the user's explicit, correct instinct: don't spend budget until the core control logic is proven sound. In parallel, begin KiCad schematic capture (FPGA header, servo header, ADC feedback header, decoupling) using an existing open-source KiCad FPGA breakout and open-source KiCad motor-driver board as reference starting points — schematic work doesn't require hardware in hand and can proceed alongside simulation.

**Phase 1 — First physical joint (Approach A scope).** Once simulation validates the control loop, purchase the FPGA dev board (Tang Nano 9K class, ~$15-40; synthesis/programming toolchain is the open-source Yosys/nextpnr-himbaechel + Gowin flow, or Sipeed's official Gowin-based tools) and route PCB rev 1 (single joint: FPGA header, driver, feedback ADC input, external power header per the Power Budget above). As reference starting points for schematic/layout conventions, use existing open-source KiCad designs — e.g. an FPGA breakout such as the `CPArti-FPGA-Development-Board` project and a motor-driver board such as the `robotont-electronics-driver-board` project (both public GitHub repos; the actual PCB merging FPGA header + driver + feedback input into one board is the original work). Submit the PCB fab order (JLCPCB) as early as possible once the schematic is validated — **note on shipping: JLCPCB standard shipping commonly runs 2-3+ weeks, not days; expedited shipping (~5-7 day transit) costs more (~$15-25 extra) but is worth budgeting for given the 1-week v1 target — factor this into the Phase 1 budget explicitly, or accept that Phase 1 hardware bring-up may land in week 2-3 if standard shipping is used.** Per the independent reviewer's mitigation, ordering early lets fab transit (whichever tier is chosen) overlap with continued HDL/testbench work rather than sitting on the critical path at the end. While the PCB is in transit, bring up the design on a breadboard/dev-board-only setup to validate real hardware behavior against the simulation. Buy servos/sensors (~$20-30) at this stage. **If time runs short in Phase 1, the fallback deliverable is open-loop PWM control on the breadboard/PCB (per the Simulation-infeasibility fallback above), not a fully-tuned closed loop.**

**Phase 2 — Multi-joint expansion (Approach B scope).** Design and verify the custom communication protocol in HDL (with its own testbench) to coordinate multiple joints from one FPGA. **Default minimal protocol (to cap scope for a first-time HDL designer): a simple synchronous shift-register daisy chain (one clock + one data line shared across joints, each joint's controller shifts in its target position) rather than an addressed/arbitrated bus.** Only attempt a more sophisticated addressed protocol if the daisy chain is working comfortably ahead of schedule. Design PCB rev 2 with multi-channel/daisy-chain connectors — **this rev requires an additional PCB fab run (~$10-20) and likely additional servo power headroom on the external supply, both explicitly incremental budget beyond the original $100 (see Constraints).** This phase is the one most likely to still be "in progress" when internship applications open — that is acceptable per the user's explicit decision; the resume and GitHub repo can show it as an actively-developed system with visible design decisions (protocol spec, rev1→rev2 changes, testbench coverage), which is itself a strong signal to reviewers.

**Phase 3 — Optional stretch (Approach C).** Only attempt the vision-in-fabric tracking pipeline after Phases 0-2 produce a working, understood multi-joint system. Treat it as a bonus differentiator, not a dependency — its absence should not block the resume-readiness of Phases 0-2.

Rationale: this sequencing honors the user's explicit "simulate before buying" requirement, preserves the independent reviewer's specific fab-timeline risk mitigation (order the PCB early off a validated schematic), keeps a real working deliverable banked at the end of each phase (so the project is never "all or nothing"), and matches the user's explicit choice to pursue the full multi-joint arm while accepting that it may reasonably be "work in progress" by application season.

## Phase 0 Simulation Results (2026-07-10)

All Phase 0 RTL and testbenches implemented and passing via Icarus Verilog (`scripts/run_sims.sh`), 4/4 testbenches green:

- `pwm_generator` — verified exact PWM duty-cycle timing at min/mid/max position against an independently-derived expected value.
- `spi_adc_reader` — verified full MCP3208-style command/response cycle against a behavioral ADC model, 4 test values including back-to-back transactions.
- `p_controller` — verified exact P-controller math, plus closed-loop convergence against a simulated lagging plant (settles to within ~15/4096 counts of target — the expected residual for pure P control, see `docs/PHASE0_NOTES.md`).
- `single_joint_controller` (full integration) — verified the complete ADC-read -> P-control -> PWM-drive loop converges end-to-end against the same plant model, plus PWM liveness.

Two real bugs found and fixed during this pass (both documented in `docs/PHASE0_NOTES.md`):
1. A one-cycle FSM race in `spi_adc_reader` where `busy` dropped one cycle before the state machine could actually accept a new `start`, silently dropping back-to-back conversion requests.
2. A signed/unsigned mixing bug in the integration testbench's pass/fail comparison (not the RTL) that made a correct -15 gap register as a wrapped-around failure.

Simulation-infeasibility fallback (per Constraints) was not needed — closed-loop control converged as designed on the first fully-debugged attempt.
